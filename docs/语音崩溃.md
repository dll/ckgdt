# 语音崩溃问题修复记录

日期：2026-06-30

## 问题现象

Windows Release 版本中，语音相关功能在以下场景出现崩溃或不可用：

- 语音登录识别成功后跳转首页时闪退。
- 语音导航执行后崩溃。
- 语音设置页第二次测试时崩溃。
- 改用 FFmpeg/dshow 后出现设备名乱码、找不到音频设备、包体过大的问题。

## 根因

日志 `build/windows/x64/runner/Release/logs/mad_init.log` 显示旧链路停在：

```text
cleanup enter reason=forceStop
```

但没有继续输出 `cleanup done`。这说明崩溃发生在录音资源释放阶段。

核心根因是 Windows 上 `record_windows` 的原生录音资源在 stop、stream cancel、dispose、页面跳转、下一轮录音启动之间存在并发释放风险。语音登录、语音导航、设置页测试都会触发“识别完成后立刻切页面或重启录音”的路径，旧 WebSocket 回调、Dart stream 回调、native recorder 释放可能交错执行，导致 Windows 原生层崩溃。

FFmpeg/dshow 虽然能绕开 `record_windows`，但引入了两个新问题：

- `ffmpeg.exe` 体积过大，不适合随 Windows 桌面端打包。
- dshow 设备名在中文 Windows 环境下容易出现乱码或匹配失败。

因此最终方案改为 Windows 原生 PCM 采集插件。

## 最终方案

### 1. Windows 原生 WASAPI PCM 采集

新增 runner 内置插件：

- `windows/runner/mad_voice_capture.h`
- `windows/runner/mad_voice_capture.cpp`

插件通过 MethodChannel `mad_voice_pcm` 暴露三个方法：

- `start`：启动 WASAPI 默认麦克风采集。
- `read`：读取已缓存的 PCM 字节。
- `stop`：停止采集并回收线程与 WASAPI 资源。

采集参数统一转换为讯飞 IAT 需要的格式：

- 采样率：16000 Hz
- 声道：mono
- 编码：PCM16 little-endian

原生侧使用默认通信麦克风，失败后回退默认控制台麦克风。启动失败会返回具体 WASAPI HRESULT，方便日志定位。

### 2. Dart 侧移除语音 FFmpeg 依赖

`lib/services/voice_pcm_capture_io.dart` 已从 FFmpeg/dshow 子进程改为调用 `mad_voice_pcm` 原生通道。

Windows 语音识别链路现在是：

```text
WASAPI -> mad_voice_pcm -> VoicePcmCapture -> VoiceService -> 讯飞 WebSocket
```

语音识别不再启动 `ffmpeg.exe`，Release 目录也不再需要 `ffmpeg.exe` 或 `ffprobe.exe`。

### 3. VoiceService 生命周期串行化

`lib/services/voice_service.dart` 调整为串行执行 start/stop/cleanup，避免录音资源被多个异步回调重复释放。

关键处理：

- 使用 `_operation` 串行化 `startListening`、`stopListening`、`forceStop`。
- 使用 `_sessionId` 让旧 WebSocket、旧音频流回调自动失效。
- 保存并取消 `_wsSub`，避免旧 WS `onDone` 干扰新会话。
- Windows 使用 `VoicePcmCapture`，非 Windows 保留 `record` 插件路径。
- `forceStop()` 改为可 await 的立即清理，确保页面跳转或下一轮录音前资源已释放。

### 4. 调用点等待 forceStop

以下语音调用点在识别完成、关闭弹窗、跳转页面或重新开始前等待 `forceStop()`：

- 登录页语音登录
- 通用语音输入按钮
- 语音导航弹窗
- 智能体聊天语音输入
- 语音设置页测试
- 课堂工具语音计时器

这解决了“识别完成后 UI 继续执行，但 native 录音器还在释放”的竞态。

### 5. 语音登录自动执行

单句模式的 `VoiceNavigationDialog` 已改为识别到非空文本后直接 `Navigator.pop(context, sentence)` 返回结果。

登录页 `_startVoiceLogin()` 收到识别结果后会自动提取学号、填充密码、等待 `VoiceService().forceStop()`，然后执行 `_login()`。因此语音登录不再需要手动点击“确定”。

## 验证结果

已执行：

```powershell
flutter analyze lib\services\voice_pcm_capture_io.dart lib\services\voice_service.dart
flutter analyze lib\presentation\widgets\voice_input_button.dart lib\presentation\pages\login\login_page.dart
flutter build windows --release
```

结果：

- 语音核心文件分析通过。
- Windows Release 构建通过。
- `build/windows/x64/runner/Release` 中未发现 `ffmpeg.exe` 或 `ffprobe.exe`。

注意：登录页和语音按钮文件仍存在项目既有的 `withOpacity` deprecation info，不影响本次语音崩溃修复。

## 预期日志

正常语音识别时应看到：

```text
using external PCM capture for Windows
native pcm capture start
native pcm capture first bytes=...
WS msg code=0 status=...
WS status=2 (end) finalText="..."
cleanup done reason=forceStop
native pcm capture stop done
```

如果原生麦克风初始化失败，日志会包含 WASAPI 步骤名和 HRESULT，例如：

```text
voice_capture_start_failed: IAudioClient::Initialize failed hr=0x...
```

## 结论

本次修复将 Windows 语音采集从 `record_windows` 和 FFmpeg/dshow 切换为内置 WASAPI PCM 采集，并通过串行生命周期管理消除了 stop/dispose/navigation/restart 之间的竞态。语音登录、语音导航、设置页测试和其它语音输入入口共享同一套稳定释放流程。
