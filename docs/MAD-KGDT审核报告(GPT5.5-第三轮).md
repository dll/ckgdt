# MAD-KGDT 审核报告（GPT5.5-第三轮）

## 一、结论

第三轮审核时，项目已经完成了第二轮报告中多数业务链路问题的代码修复：作品评分回传、平时成绩同步与课程隔离、实验提交后的 AI 待审核草稿、内层 Tab 注册表反向漂移测试、归档结课去重和必备材料勾选校验都已经有明确实现和测试支撑。

本轮发现并修复了一个新的工具链兼容阻塞：在 Flutter 3.35.1 下，`ThemeData` 的 `cardTheme/dialogTheme/tabBarTheme` 需要使用 `CardThemeData/DialogThemeData/TabBarThemeData`。修复后全量测试恢复通过。

当前仍不建议直接声明正式发布包已闭合。核心原因不是业务源码编译错误，而是 Windows release 构建仍被正在运行的 release 程序占用，链接阶段无法覆盖 exe。关闭该进程后必须重新执行 `flutter build windows --release`。如果目标是校内继续试用，当前版本可以作为候选版本继续验收；如果目标是公开发布，还需要完成发布构建复测、凭据治理和一次跨设备端到端验收。

## 六、关键发现

### P0-1 Windows release 发布构建仍未闭合

位置：Windows 发布链路。

现象：`flutter build windows --release` 在链接阶段失败，无法打开 release 目录中的 `课程图谱与数字孪生v2.1.0.exe`。本机存在同路径运行进程 PID `28120`。

影响：不能对外宣称 release 包已成功重新构建。第二轮是 WebView2 DLL 被运行中应用锁定，本轮是 exe 自身被运行中应用锁定，本质问题相同：构建前没有关闭旧 release 程序。

建议：

1. 关闭 PID `28120` 对应的正在运行程序。
2. 重新执行 `flutter build windows --release`。
3. 构建通过后做最小冒烟：启动、登录、学习、实验、考核、作品、达成、归档。
4. 发布脚本增加构建前进程检测，发现 release exe 正在运行时先提示教师关闭程序。

### P1-1 发布凭据治理仍是公开发布风险

位置：AI 配置、Gitee 同步配置、初始化默认配置。

现状：项目仍保留默认 AI key 和共享同步 token 设计。考虑到用户要求新用户开箱即用，本报告不建议简单删除默认能力。

影响：校内课堂试用可以接受，但公开仓库、公开安装包或对外 release 会扩散凭据。同步 token 如果有写权限，还会带来学生数据仓库安全风险。

建议：

1. 将默认凭据明确区分为“课堂试用版”。
2. 限制额度、权限范围和有效期，保留轮换/禁用开关。
3. 对公开发布包改用服务端代理或首次启动配置向导。

### P1-2 Flutter 3.35 analyzer info 大幅增加

位置：全项目。

现象：`flutter analyze --no-pub` 返回 1593 个 info，0 warning，0 error。主要是 `withOpacity` 与颜色通道 API deprecated。

影响：当前不阻断测试和构建编译，但维护成本明显增加。未来 Flutter 再升级时，部分 deprecated API 可能继续收紧。

建议：

1. 不要一次性机械改完整个项目，优先改公共组件、主题、达成、归档、作品这类高频页面。
2. 对 `Color` 兼容层统一封装，避免每个页面分散处理。
3. 把 analyzer warning/error 作为硬门禁，info 作为逐步清理指标。

### P2-1 归档班级过滤已改善，但仍存在历史特例代码

位置：`ActiveStudentScope`、`AchievementDao`、`OrdinaryScoreDao`、`ClassDao.getProjectGroups()`。

现状：

- 当前成绩统计、达成平台聚合、平时成绩快照使用 `ActiveStudentScope`，排除只属于归档班级的学生。
- 相关测试已覆盖“平台聚合成绩生成三张分项表并排除归档班级学生”和“平时成绩快照排除只属于归档班级的学生”。
- `ClassDao.getProjectGroups()` 仍保留归档“计科22”的历史真实分组特例。

影响：统计链路已经走向正确，但历史特例如果被新的教学页面误用，仍可能让教师看到归档班级材料。

建议：保留历史查看入口可以，但要用页面权限和调用路径隔离，确保“当前教学、成绩统计、学情分析、学情预警”不调用历史特例。

### P2-2 跨设备端到端验收仍需要补

位置：作品、平时成绩、同步服务。

现状：代码链路和单元测试已经明显改善，但以下场景仍需要真实双端验收：

1. 学生提交作品视频，教师端下载同步后播放、评论、点赞、评分。
2. 教师 AI 批阅作品后，学生端同步看到最终分数和评语。
3. 学生使用 AI 自主学习、推荐视频收藏、课件学习计时后，教师端平时成绩同步一致。
4. 教师端达成成绩管理从平台聚合生成平时、实验、考核三表，再计算达成度。

这些不是当前单测能完全替代的文件附件和跨设备同步问题。

## 七、模块审核

### 7.1 首页与课程目标

课程目标模块已成为教师、学生、管理员可见的公共入口，并补充课程目标、毕业要求支撑、考核方式和达成关系。对当前《移动应用开发》课程是可用的。

剩余建议：继续减少静态课程内容，优先从当前课程大纲、课程目标表或归档大纲解析结果读取，静态内容仅作为移动应用开发默认模板。

### 7.2 学习与平时成绩

教师端“平时成绩”已经覆盖课堂表现 20、期间测验 30、课外学习 50，并支持指标设置、达成度批次同步、学生明细表、学号/姓名筛选、总分升降序。学生端学习成绩入口也已与该指标体系对齐。

代码复核结果：

- 课堂表现按课程过滤点名、签到、课堂回答。
- 期间测验读取当前课程测验成绩。
- 课外学习读取课件学习、AI 自主学习、扩展/推荐学习。
- 学生列表通过 `ActiveStudentScope` 排除只属于归档班级的学生。

判断：第二轮指出的同步和课程隔离问题已经有明确修复。后续重点是跨设备真实数据一致性。

### 7.3 实验

实验提交后 AI 草稿链路已经接入：学生提交时先写提交记录，再根据设置调用 AI 批阅生成 pending 草稿，教师审核通过后再形成正式批阅。AI 不可用或低于通过线时会删除提交和 pending 草稿，避免出现半提交状态。

实验成绩也纳入达成平台聚合，按当前课程实验任务最多映射 7 个实验分项。全量测试中的达成流测试已经覆盖无实验课程、6 实验模板、7 实验模板等情况。

### 7.4 考核

考核终稿审核打印流程相较第二轮已经更完整，方向符合“不是四个子报告简单整合，而是最后大作业报告涵盖子报告”的需求。封面、批语、成绩、审核通过/不通过、打印标识等状态机仍建议做一次 UI 端到端脚本或手工验收。

本轮没有发现新的源码级阻塞。

### 7.5 作品

作品链路已经围绕用户截图中的矛盾点做了实质修复：

- 提交标识和作品归属统一使用 `user_id`。
- 播放前会尝试通过同步服务补下载学生作品附件。
- 教师/AI 评分后触发通知和学生数据上传。
- 同步包包含学生作为评分人的互评，以及该学生作品上的教师/admin 评分。
- 导入端按作品自然键重映射 `work_id`，避免直接复用跨设备自增 ID。

判断：代码层已经解决第二轮“有提交、有批阅、播放/分数不一致”的主要根因。仍需真实附件同步验收，尤其是视频文件名、路径、Gitee 仓库目录和本地缓存路径。

### 7.6 达成与成绩管理

达成模块现在能从平台统一聚合：

- 平时成绩：课堂表现、期间测验、课外学习。
- 实验成绩：实验提交评分按任务映射。
- 考核成绩：考核报告、项目/小组/个人/答辩、作品评分等数据合并。

导入成绩和平台聚合成绩的结构趋于一致，能够直接填充用于达成度计算的三张分项表。测试已覆盖平台聚合排除归档班级学生。

剩余建议：继续补“从平台聚合到 Word/Excel 达成报告”的端到端测试，尤其是截图、图表和学校模板格式。

### 7.7 归档

期初、期中、期末、结课归档已经形成统一流水线：

- 多格式导入：MHTML/HTML、DOCX、MD、PDF、XLSX 等按材料类型解析或作为原件引用。
- 生成：可从模板/源文件/系统数据生成 Markdown 或归档件。
- 预览：可查看内容或原件提示。
- 审核：结构化审核结果可保存为文档。
- 打印/归档：桌面端生成 docx/pdf/zip，Pandoc 可用时继承模板样式，不可用时回退。
- 结课：勾选已形成材料后打包，必备材料必须进入结课包，重复确认表会清理。

本轮全量测试覆盖了期初 E2E、期初模板解析、期中核查、期末 00-12 编号材料解析、Pandoc 转换、审核结果 Markdown 等。归档方向已经比第二轮更稳定。

### 7.8 直播与跨设备

本轮没有对答辩直播做实机网络矩阵。第一轮/第二轮关于局域网端点鉴权、UDP 发现、HTTP/MJPEG 明文访问的风险仍成立。当前有 MJPEG 分片解析基础测试，但不能替代真实网络和多设备验收。

## 九、最终判断

第三轮相较第二轮，项目的主业务链路继续收敛：测试规模增加且全量通过，归档期初/期中/期末/结课已有统一流水线，平时成绩和达成聚合的归档班级过滤已有测试守护，作品评分回传的数据链路也已经从字段、同步包和导入端三个层面补齐。

当前最大剩余问题已经从“功能链路明显缺口”转移为“发布构建和真实环境验收”。关闭正在运行的 release 程序并重跑 Windows 构建，是下一步必须先做的发布门禁。业务上最需要验收的是作品视频附件跨设备播放和教师评分回传学生端，其次是平时成绩多数据源同步后与达成聚合保持一致。
