## 一、项目由来

### 1.1 背景

《移动应用开发》课程是软件工程专业的核心课程，涉及 Android（Java/Kotlin）、iOS（Swift）、Flutter、React Native、微信小程序、华为 HarmonyOS（ArkUI）六大技术栈。传统教学面临以下困境：

1. **知识点散**：六大技术栈并行，学生难以构建知识体系
2. **迭代快**：每年都有新技术框架出现，教材跟不上
3. **实验链路长**：从环境搭建到真机调试，每个实验 3-6 小时
4. **批阅工作量巨大**：88 人班级 × 6 个实验 × 多人分工 = 528 份报告
5. **达成度计算手工作**：工程教育认证（OBE）需要三维成绩分析
6. **缺少 AI 辅助**：学生课后无答疑，教师缺乏智能批阅工具

### 1.2 立项

**2026 年 4 月**，项目正式启动，目标是用 Flutter 构建一个**全栈式教学平台**，把"教—学—练—评—管"五大场景全部装进一个应用。技术选型：

| 层面 | 选型 | 理由 |
|------|------|------|
| 跨平台 UI | Flutter 3 + Material Design 3 | 一套代码覆盖 Android/Windows/Web/HarmonyOS |
| 本地数据库 | sqflite（SQLite） | 离线可用，无需后端服务器 |
| AI 服务 | DeepSeek / 智谱 GLM-4 | 多 Provider 切换，API Key 存 DB |
| 同步方案 | Gitee 仓库 JSON 文件 | 无服务器架构，适合学校弱网络环境 |
| 智能体框架 | BaseAgent + AgentRegistry | 领域专精 + 统一抽象 |
| 图谱绘制 | CustomPainter + InteractiveViewer | 自定义力导向布局 |

### 1.3 原始代码基础

项目在启动前已有一个初步的 Flutter 教学原型（约 143,000 行），包含：
- 基本的知识图谱浏览（6 大类 32 个图谱）
- 章节测验系统（52 道预置题目）
- 实验任务管理
- 学生/教师双角色导航
- SQLite 数据库（59 张表）
- Gitee 数据同步雏形

但存在明显的工程问题：
- `catch (_)` 静默吞错 356 处
- 测试覆盖率 < 1%（仅 7 个测试文件，21 用例）
- 无 CI/CD
- 24 个 AI 智能体只有并联、无协作（无 Orchestrator）
- Prompt 硬编码在 .dart 代码中
- RAG 只有关键词匹配、无向量检索
- 巨型文件多达 6-7 个（如 assessment_page.dart 6090 行）

### 3.2 RAG 检索增强

```dart
启动时 RagBootstrapService.ensureIndexed() 后台 unawaited 调用：
  ├── concepts → indexDocument → rag_embeddings
  ├── resources → indexDocument
  └── questions → indexDocument

BaseAgent.buildRagPrompt：
  if (rag_embeddings.count > 0):
    retrieveContextVector  ← embedding + 余弦 top-K
  else:
    retrieveContext  ← TF-IDF 兜底

EmbeddingService LRU 缓存 maxSize=64：
  按 (provider|model|text) 复合 key
```

### 3.3 多 Agent 协作（Orchestrator）

```dart
教师批阅页打开「增强批阅」Switch → 走子路由：

  grading_agent    ← 主批阅（功能/代码/报告/创新四维评分）
```

## 七、关键架构决策

### ADR 1：单体 Flutter 而非 BFF + Native
- **选择**：所有逻辑在 Flutter 应用内完成
- **理由**：教学场景部署难度优先，Flutter 4 端复用 ≥ 90% 代码
- **代价**：~148,000 行单体代码，部分文件过大

### ADR 2：Gitee 作消息总线
- **选择**：通过 Gitee 仓库 JSON 文件实现师生同步
- **理由**：学校机房网络受限不能开服务器，Gitee 当数据库 + 消息队列
- **代价**：全班共用 Gitee Token，恶意学生可破坏数据

### ADR 3：多 Agent 矩阵而非单一对话
- **选择**：24 个专精 Agent，每种教学场景独立 prompt + 工具
- **理由**：批阅/答疑/生课各需要不同的知识和能力
- **代价**：Agent 间串联需要 Orchestrator 编排

### ADR 4：part/part of 拆分巨型文件
- **选择**：使用 Dart part 机制拆分大页面
- **约束**：part 只能拆分顶层 class，不能拆 State 内成员
- **未拆**：courseware_workshop_page 3811 行（需 helper class 重构，搁置）
